### 3. Implementation Details
- **Data layer isolation**: All UI reads/writes go through hooks in
  `features/billing/hooks.ts`, which call `useBillingStore`. Swapping to a
  real backend means rewriting only `store.ts`.
- **Dashboard**: revenue (paid), outstanding (sent+overdue), overdue totals,
  recent activity table.
- **Invoices**: list with search + status filter, create/edit form with
  dynamic line items + live totals, detail view, CSV export/import.
- **Customers**: grid + add/edit dialogs + search.
- **Auth**: login/register UI with Zod validation (mock store).
- **Routing**: TanStack Router file-based, `__root.tsx` provides
  `SidebarProvider` + `Outlet`.

### 4. Technical Decisions
- **Zustand** over Redux: minimal boilerplate, easy to swap for API later.
- **Zod** schemas as single source of truth for types and validation.
- **Hook-only consumption**: prevents components from coupling to the store
  shape, enabling backend swap without UI refactors.
- **`useShallow`** for object selectors (Zustand v5 requirement) to avoid
  infinite re-renders.

### 5. Code Logic Explanation
- `invoiceTotals(items, taxRate)` → `{ subtotal, tax, total }`.
- `useFilteredInvoices(q, status)` → memoized filter by number/customer/status.
- `useDashboardStats()` → aggregates totals by status.
- `useBillingStore` → seed customers (`c1-c8`) + 10 seed invoices; CRUD ops
  generate ids via `uid()` and invoice numbers `INV-1000+n`.
